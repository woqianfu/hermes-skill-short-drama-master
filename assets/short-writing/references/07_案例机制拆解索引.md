# 短剧案例机制拆解索引

## 1. 拆解原则

案例只迁移机制，不复制剧情、人名、台词和具体设定。拆解对象是题材承诺、第一屏、人物功能位、前10集卡点、追更点和中段规则变化。

## 2. 案例拆解表

```text
案例名：

第一集开局：
前三集钩子：
第一个追更点：
人物功能位：
可迁移机制：
必须替换的内容：
可用于我项目的原创方案：
```

## 3. 当前机制索引

结构化机制索引见 `processed/case_mechanism_index.csv`。该文件只记录分类、机制、开场模型、前10集模型和迁移说明，不收录完整剧本原文。


## 案例机制拆解维度

| 维度 | 要记录什么 | 不能记录什么 |
| --- | --- | --- |
| 题材承诺 | 观众等的情绪 | 剧情复述 |
| 第一屏 | 谁压谁、凭什么压 | 完整台词 |
| 人物功能 | 主角、反派、误判者、助攻各自作用 | 角色履历流水账 |
| 前10集 | 每集结尾问题和第一个追更点 | 逐场照搬 |
| 证据物件 | 合同、鉴定、录音、病历等功能 | 具体桥段顺序 |
| 中段升级 | 换目标、换敌人、换规则 | 原作全部分集 |

## 机制迁移三步

```text
1. 抽功能：公开羞辱 / 证据将揭 / 身份错认 / 关系站错 / 资源封锁。
2. 换皮肉：换人物关系、行业、场景、证据物件和反派筹码。
3. 重新卡点：把机制放进自己的第1集、第3集、第7集、第10集。
```

`processed/case_mechanism_index.csv` 保存 875 条样本机制条目。它只用于查询题材、样本数、开场模型、前10集模型和迁移提示，不作为原文库使用。

## 查询案例的标准流程

```text
1. 先定主赛道：例如追妻火葬场。
2. 在索引中筛同题材样本，读 opening_model 和 first10_model。
3. 把机制拆成：压迫、误判、证据、反击、钩子。
4. 换掉人物身份、关系旧账、证据物件和场景。
5. 放进自己的第1集、第3集、第7集、第10集。
6. 自检：如果还像原样本，说明只换皮没换机制。
```


## 拆样本禁区

```text
不要复制完整剧本原文。
不要复制人名、台词、连续桥段和具体分集。
不要把同一个样本的结构照搬成新项目。
不要把案例索引当素材库直接续写。
只允许抽取机制、卡点、功能位和可替换场景。
```

## 案例机制拆解、对标边界与原创化转译

## Agent 角色设定

案例拆解与原创化风控型 Agent，负责从案例中抽机制，不抽原文和连续桥段。

工作目标：
- 把案例拆成类型承诺、第一屏压力、主压迫、核心秘密、前10集节奏和中段升级。
- 区分可借鉴机制和必须替换元素。
- 对标时只输出功能、节奏、压力和反转用途。
- 把单个案例与至少两个同类案例交叉比对，避免单剧复刻。
- 最终新项目必须更换人物关系、证据物件、职业场景、关键反转和对白表达。

禁止做法：
- 不要用抽象评价代替场景级判断。
- 不要把类型公式当成固定剧情模板。
- 不要复用本地完整剧本的人物、场景、对白或连续桥段。
- 不要在缺少信息时停在追问；先给合理假设并继续输出可执行版本。
- 不要让正文变成小说梗概；正文必须能被演员和导演直接理解。

逐项诊断写法：
### 诊断 1：只列剧名

- 诊断信号：没有说明可借鉴什么。
- 深层原因：短剧的压力、行动、证据、关系或钩子没有承担应有功能。

- 输出要求：指出原材料中 1-3 个具体位置，再给出可直接替换的结构或场景写法。

## 常见失败

| 失败类型 | 后果 | 修正 |
| --- | --- | --- |
| 只讲理论 | 用户拿不到可写内容。 | 每段理论后补一个结构动作或输出字段。 |
| 直接写正文 | 还没确定类型和第一屏。 | 先做任务判断和类型承诺。 |
| 案例复刻 | 新稿像某一部剧。 | 至少混合三个机制，并列必须替换项。 |
| 压力不可见 | 观众不知道主角为何必须行动。 | 改成公开场、可见损失、旁观者站队。 |
| 反击工具缺失 | 主角只靠忍耐和解释。 | 补证据、身份、资源、技能、孩子线索或规则漏洞。 |
| 尾钩模糊 | 每集结尾没有下一集问题。 | 用证据、来人、电话、身份露角或关系选择收尾。 |
| 对白说明化 | 人物替作者解释设定。 | 让信息成为争夺物。 |
| 中段重复 | 同类争吵循环。 | 升级压力来源和裁判层级。 |
| 结尾无清算 | 真相公开但情感债没还。 | 补承认、补偿、放权、公开承担。 |
| 格式混乱 | 正文像小说或分镜秒表。 | 统一剧本格式。 |
